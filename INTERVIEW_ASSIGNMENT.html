MenuData

Software Engineering Interview Assignment

Take-Home

Overview

Welcome to the software engineering take-home assignment. In this exercise, you will build an AI-powered marketing insights agent that analyzes restaurant menu data to discover flavor trends, ingredient combinations, and innovation opportunities.

Your Mission: Build an intelligent agent that can answer questions about menu trends, flavor combinations, and market opportunities using a Neo4j graph database and an LLM. Then create an evaluation suite using Braintrust to measure the agent's performance.

Provided Data

We're providing you with a dataset of ~50,000 menu items from 500 NYC restaurants, including ingredient mappings and generalized menu item (GMI) classifications.

menu_items.csv Menu items with restaurant info 50,000 rows
restaurants.csv Restaurant locations & types 500 rows
ingredients.csv Ingredient taxonomy 159 rows
gmis.csv Generalized Menu Items 55 rows
item_ingredients.csv Item-to-ingredient mappings 151,163 rows

Data Schema

File Key Columns
menu_items.csv item_id, menu_item_name, description, price, restaurant_id, restaurant_name, city, state, restaurant_type, chain_name, gmi_name
restaurants.csv restaurant_id, business_name, city, state, restaurant_type, chain_name, cuisine
ingredients.csv ingredient_id, ingredient_name, category (protein, vegetable, dairy, grain, sauce, spice, fruit, sweetener, oil, nut)
gmis.csv gmi_id, gmi_name, category (Entree, Appetizer, Side, Dessert, Beverage, Soup)
item_ingredients.csv item_id, ingredient_id, ingredient_name, ingredient_category

Part 1: Neo4j Graph Model Design

Design and implement a Neo4j graph database model to represent the menu data. Your model should enable efficient queries about flavor combinations, ingredient relationships, and menu item patterns.

Starter Ideas (Build on These)

(:Restaurant)-[:SERVES]->(:MenuItem)
(:MenuItem)-[:CONTAINS]->(:Ingredient)
(:MenuItem)-[:IS_TYPE]->(:GMI)
(:Ingredient)-[:CATEGORY]->(:IngredientCategory)
(:Restaurant)-[:LOCATED_IN]->(:Neighborhood)
(:Restaurant)-[:CUISINE_TYPE]->(:Cuisine)

Consider adding relationships such as:

  • Ingredient Co-occurrence: (:Ingredient)-[:PAIRS_WITH {frequency: N}]->(:Ingredient)
  • Flavor Profiles: Group ingredients by flavor characteristics (spicy, sweet, savory, umami)
  • Price Tiers: Connect items to price ranges for market positioning analysis
  • Chain Relationships: Model multi-location restaurant chains

Part 2: Build Your Agent

Choose one of the following agent types to implement. Your agent should use the Neo4j graph database and an LLM to generate insights.

Agent Option A: Flavor Innovation Agent

An agent that discovers emerging flavor combinations and innovation opportunities by analyzing ingredient co-occurrence patterns across different cuisines and restaurant types.

Example Questions the Agent Should Answer:

  • "What unexpected ingredient combinations are appearing in Brooklyn fine dining?"
  • "Which Asian fusion flavor pairings are crossing over to American cuisine?"
  • "Find emerging spice combinations in fast-casual Mexican restaurants"
  • "What ingredients pair well with yuzu that aren't commonly used?"
  • "Identify white-space opportunities: ingredients popular in Japanese cuisine missing from Korean menus"

Agent Option B: Market Intelligence Agent

An agent that generates competitive intelligence and market positioning insights by analyzing menu pricing, item distribution, and regional trends.

Example Questions the Agent Should Answer:

  • "How do burger prices vary between QSR and casual dining in Manhattan vs Brooklyn?"
  • "Which GMI categories are over-indexed in the East Village compared to Midtown?"
  • "What menu items do successful Italian restaurants have that struggling ones don't?"
  • "Find price gaps: dishes that could be offered at premium prices based on ingredient quality"
  • "Compare ingredient sophistication between chain restaurants and independents"

Technical Requirements

  • Use Neo4j as your graph database (local Docker instance or Neo4j Aura free tier)
  • Use any LLM provider (OpenAI, Anthropic, etc.) for the agent reasoning
  • The agent should translate natural language questions into Cypher queries
  • The agent should synthesize query results into actionable insights
  • Include error handling for malformed queries and edge cases

Part 3: Evaluation Suite with Braintrust

Create an evaluation suite using Braintrust to measure your agent's performance. This is critical for iterating on agent quality.

Evaluation Dimensions

Dimension Description Weight
Accuracy Does the answer correctly reflect the data? Are statistics accurate? 30%
Relevance Does the response directly answer the question asked? 25%
Creativity Does the agent surface non-obvious insights? Make unexpected connections? 25%
Latency Response time (aim for < 10 seconds for typical queries) 20%

Test Cases to Include

Create at least 15 test cases covering:

  • Simple lookups (e.g., "What are the most common ingredients in ramen?")
  • Comparative queries (e.g., "Compare pizza toppings between Italian and American restaurants")
  • Trend analysis (e.g., "What flavors are trending in desserts?")
  • Cross-cuisine insights (e.g., "Find fusion opportunities between Thai and Mexican")
  • Edge cases (ambiguous queries, empty results, very broad questions)

Deliverables

  1. Neo4j Data Model - Cypher scripts to create your graph schema and load the data
  2. Agent Implementation - Python code for your chosen agent with clear documentation
  3. Braintrust Evaluation - Evaluation suite with 15+ test cases and scoring functions
  4. README - Setup instructions, architecture decisions, and sample outputs
  5. Demo Queries - 5 interesting queries with their outputs showcasing your agent's capabilities

How We'll Evaluate Your Submission

  • Graph Model Design (25%) - Is the model well-suited for the queries? Efficient? Extensible?
  • Agent Quality (30%) - Does it produce useful, accurate insights? Handle edge cases?
  • Evaluation Suite (25%) - Are the test cases comprehensive? Are the scoring functions meaningful?
  • Code Quality (20%) - Clean, readable, well-documented code with proper error handling
